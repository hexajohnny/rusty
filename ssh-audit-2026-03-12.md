# Rusty SSH/SFTP Audit

Date: 2026-03-12

Scope: current SSH transport, authentication, terminal I/O, SFTP/file-manager flow, transfer manager, and persistence behavior in this repo.

Validation run:
- `cargo test` passed.
- `cargo clippy --all-targets -- -D warnings` passed.
- No live server smoke test was run, so transport/auth findings below are based on code inspection rather than an end-to-end host matrix.

## Confirmed Bugs And High-Value Issues

- High: remote upload finalization is delete-then-rename, so a failed rename can destroy the existing destination file.
  `replace_remote_file` removes `final_path` first and only then renames the temp file into place. If the rename fails after the delete because of permissions, server behavior, cross-filesystem limitations, or a transport error, the original file is already gone and the upload has caused data loss.
  Evidence: `src/ssh.rs:930`, `src/ssh.rs:936`, `src/ssh.rs:941`, `src/ssh.rs:1124`, `src/ssh.rs:1258`.

- High: transfer resume integrity is size-only, which can silently corrupt resumed downloads and uploads.
  Download resume trusts only `local_path.rusty-part` length versus current remote length. Upload resume trusts only `remote_path.rusty-part` length versus current local length. There is no checksum, mtime, inode/fingerprint, or session token recorded. Reusing the same path after the source changed can splice stale partial bytes into a new file while still reporting success.
  Evidence: `src/ssh.rs:885`, `src/ssh.rs:889`, `src/ssh.rs:974`, `src/ssh.rs:1005`, `src/ssh.rs:1111`, `src/ssh.rs:1115`, `src/ssh.rs:1153`.

- Medium: transfer history snapshots full `ConnectionSettings` into every job, which creates secret sprawl and stale-retry behavior.
  Each persisted transfer stores a full credential snapshot, then retry uses that stored snapshot rather than re-resolving the current profile. That means password/key changes made after a failed transfer may not be used by a later detached retry, and the config accumulates multiple encrypted copies of the same secrets.
  Evidence: `src/config.rs:153`, `src/config.rs:157`, `src/app/10_impl_core.rs:514`, `src/app/10_impl_core.rs:518`, `src/app/10_impl_core.rs:1629`, `src/app/10_impl_core.rs:1710`.

- Medium: "Delete + Remove" does not actually clean up partial download temp files, and uploads have no equivalent remote cleanup path.
  Download removal only deletes `local_path`, but partial downloads are stored at `local_path.rusty-part`. Partial upload data lives remotely at `remote_path.rusty-part`, and removing upload history does not attempt remote cleanup at all. This leaves orphaned temp files behind while the UI wording strongly implies cleanup.
  Evidence: `src/app/22_impl_downloads_window.rs:27`, `src/app/22_impl_downloads_window.rs:40`, `src/app/10_impl_core.rs:1793`, `src/app/10_impl_core.rs:1809`, `src/ssh.rs:885`, `src/ssh.rs:889`.

- Medium: resumed upload speed reporting is wrong.
  Resumed uploads initialize `uploaded_bytes` from the already-present remote temp file, then divide that total by the elapsed time of the current attempt. The first progress updates can therefore report impossible speeds because previously uploaded bytes are counted as if they were sent during the current run.
  Evidence: `src/ssh.rs:1112`, `src/ssh.rs:1153`, `src/ssh.rs:1212`, `src/ssh.rs:1236`, `src/ssh.rs:1244`.

- Medium: agent authentication can fail on servers with low `MaxAuthTries` because Rusty blindly offers every agent identity.
  `authenticate_via_agent` iterates every key returned by the agent and there is no per-profile way to pin a specific agent identity, disable agent auth, or cap attempts. Users with many loaded keys can fail before the correct identity is even tried.
  Evidence: `src/ssh.rs:1548`, `src/ssh.rs:1561`, `src/ssh.rs:1567`, `src/model.rs:4`.

- Medium: detached retries are not truly self-sufficient.
  If the source terminal is gone, retry falls back to a detached session, but detached auth only works with already-saved credentials and non-interactive host trust. MFA prompts, password prompts, key-passphrase prompts, and first-use host trust decisions cannot be completed in detached mode, so retries can fail even though the original live session worked.
  Evidence: `src/ssh.rs:1487`, `src/ssh.rs:1510`, `src/ssh.rs:1592`, `src/ssh.rs:1721`, `src/ssh.rs:1946`, `src/ssh.rs:1967`, `src/app/10_impl_core.rs:1702`.

- Medium: terminal keyboard coverage is too narrow for serious SSH/TUI use.
  Only basic keys are translated. There is no mapping for F1-F12, Shift-Tab/backtab, Alt/meta escape sequences, or modified arrows/home/end/page keys. That will break common workflows in `vim`, `tmux`, `less`, readline, Midnight Commander, and many ncurses apps.
  Evidence: `src/app/11_impl_terminal_helpers.rs:240`, `src/app/11_impl_terminal_helpers.rs:254`, `src/app/11_impl_terminal_helpers.rs:269`, `src/app/11_impl_terminal_helpers.rs:704`, `src/app/11_impl_terminal_helpers.rs:766`.

- Medium: remote mouse protocol support is incomplete.
  Right-click is always consumed as local paste, wheel events always drive local scrollback, and the remote only gets a primary-button press/release pair. Drag/move reporting and genuine secondary/middle-button interaction are effectively unavailable, which breaks mouse-heavy TUIs.
  Evidence: `src/app/11_impl_terminal_helpers.rs:277`, `src/app/11_impl_terminal_helpers.rs:301`, `src/app/11_impl_terminal_helpers.rs:367`, `src/app/11_impl_terminal_helpers.rs:428`, `src/app/11_impl_terminal_helpers.rs:625`, `src/app/11_impl_terminal_helpers.rs:648`.

- Medium: remote path handling is POSIX-centric and likely wrong for Windows SFTP roots.
  Parent resolution only understands `/` semantics. Paths such as `C:/` or drive-like canonical roots can collapse to `.` or `C:` instead of a stable root, which is a real compatibility risk against Windows OpenSSH or other Windows SFTP servers.
  Evidence: `src/app/10_impl_core.rs:1066`, `src/app/10_impl_core.rs:1084`, `src/app/10_impl_core.rs:1097`, `src/app/10_impl_core.rs:1323`.

- Medium: file-manager semantics are too thin for directories and non-regular files.
  Directory download is disabled, upload only opens a file picker, delete uses non-recursive `remove_dir`, and the SFTP metadata model only distinguishes "dir" vs "not dir". Symlinks, permissions, owners, and recursive directory operations are not modeled well enough for a dependable remote file manager.
  Evidence: `src/app/21_impl_file_manager_view.rs:127`, `src/app/21_impl_file_manager_view.rs:132`, `src/app/10_impl_core.rs:1414`, `src/ssh.rs:243`, `src/ssh.rs:251`, `src/ssh.rs:693`.

## Engineering And Product Considerations

- The live shell auth path and detached transfer auth path are duplicated implementations.
  `open_authenticated_session_for_transfer` and `run_shell_async` both contain their own auth sequencing, capability probing, and fallback logic. That duplication already creates behavior drift, especially around prompting, host-key UX, and retry semantics.
  Evidence: `src/ssh.rs:1592`, `src/ssh.rs:1725`, `src/ssh.rs:2145`, `src/ssh.rs:2517`.

- SFTP/file-manager lifecycle is coupled to a live terminal worker instead of being a first-class standalone session.
  File-manager attach and transfer start both depend on a source terminal tile and its worker channel. That increases coupling, makes file operations depend on shell-tab health, and leaves detached transfers as a partial fallback rather than a clean primary path.
  Evidence: `src/app/10_impl_core.rs:1113`, `src/app/10_impl_core.rs:1131`, `src/app/10_impl_core.rs:1396`, `src/app/10_impl_core.rs:1511`.

- Auth heuristics rely on brittle string matching.
  Cached password auto-fill only triggers when a keyboard-interactive prompt "looks like password", and private-key re-prompt depends on matching error text for passphrase-related words. Small upstream wording changes can silently degrade the UX.
  Evidence: `src/ssh.rs:188`, `src/ssh.rs:193`, `src/ssh.rs:1396`, `src/ssh.rs:2284`.

- The shell model is fixed to `request_shell` plus `TERM=xterm-256color`.
  There is no way to request `exec`, pick a different terminal type, or provide per-profile startup environment. That is a real limitation for servers that expect explicit commands, alternate terms, or environment setup.
  Evidence: `src/model.rs:4`, `src/ssh.rs:2526`, `src/ssh.rs:2533`, `src/ssh.rs:2537`.

- Automated coverage is still too light for the critical SSH behaviors.
  The current tests are unit-level and there is no `tests/` integration suite for real SSH servers, host-key rotation, keyboard-interactive MFA, transfer resume integrity, or detached retry behavior. For this surface area, that is a real regression risk.
  Evidence: no `tests/` directory in the repo root; current `cargo test` coverage is limited to unit tests.

## Must-Have Missing SSH And SFTP Features

- `~/.ssh/config` import and resolution.
  At minimum: `Host` aliases, `IdentityFile`, `IdentityAgent`, `ProxyJump`/`ProxyCommand`, `UserKnownHostsFile`, `StrictHostKeyChecking`, and `Port`/`User` overrides. Current connection settings only expose host, port, username, password, one key path, and one key passphrase.
  Evidence: `src/model.rs:4`.

- Multiple identities and explicit auth-policy controls.
  The app needs per-profile control over auth order and auth sources: key-only, agent-only, password disabled, selected agent identity, multiple key files, and ideally OpenSSH certificate support. This is the clean fix for crowded-agent failures and stricter enterprise setups.
  Evidence: `src/model.rs:4`, `src/ssh.rs:1548`.

- ProxyJump / bastion support.
  This is table-stakes for many real environments. Right now connections are direct TCP only.
  Evidence: `src/ssh.rs:1605`, `src/ssh.rs:2163`.

- Port forwarding.
  Local, remote, and dynamic forwarding are core SSH features and there is no trace of them in the current connection model or runtime.
  Evidence: `src/model.rs:4`, `src/ssh.rs:2520`.

- Agent forwarding.
  Local agent auth is supported, but forwarding the agent to the remote session is not. That is a must-have in multi-hop workflows.
  Evidence: `src/ssh.rs:1522`, `src/ssh.rs:2537`.

- Per-profile startup command, working directory, environment, and configurable `TERM`.
  Many users need "connect and run X", or a reliable initial shell command, or a specific terminal type. The current session path is shell-only and hard-coded to `xterm-256color`.
  Evidence: `src/model.rs:4`, `src/ssh.rs:2528`, `src/ssh.rs:2533`, `src/ssh.rs:2537`.

- Standalone SFTP sessions and transfers that do not depend on a live terminal tab.
  File-manager panes and transfers should be able to establish and own their own authenticated SSH/SFTP session cleanly, with shared auth plumbing but separate lifecycle.
  Evidence: `src/app/10_impl_core.rs:1113`, `src/app/10_impl_core.rs:1131`, `src/app/10_impl_core.rs:1396`.

- Recursive directory upload/download and better file-manager ergonomics.
  Must-have items: folder picker for upload, recursive directory transfer, multi-select, drag-and-drop upload, and better directory delete/copy/move semantics.
  Evidence: `src/app/21_impl_file_manager_view.rs:127`, `src/app/10_impl_core.rs:1414`.

- Symlink, permission, and ownership support in SFTP.
  The UI and transfer model should expose symlink detection plus `chmod`, `chown`, and permission/owner display where the server allows it.
  Evidence: `src/ssh.rs:243`, `src/ssh.rs:251`, `src/ssh.rs:650`.

- Known-hosts management beyond the one-time trust prompt.
  A usable SSH client needs a place to inspect, remove, replace, and explain pinned host keys, plus policy choices like accept-new vs strict. The current UI only handles connect-time prompts.
  Evidence: `src/ssh.rs:1794`, `src/app/12_impl_settings.rs:1494`.

- Stronger transfer integrity and conflict policy.
  Must-have items: checksum or mtime validation for resume, conflict prompts or rename-on-conflict behavior, remote temp cleanup controls, and a truly atomic/extension-aware remote finalize path.
  Evidence: `src/ssh.rs:930`, `src/ssh.rs:974`, `src/ssh.rs:1115`.
